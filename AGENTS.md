# AGENTS.md

## 기본 원칙
0. `UTF-8`을 항상 적용한다.
1. 항상 `공식문서` 기준으로 판단한다.
2. 항상 이 프로젝트의 `기존 코드`와 `구조`를 최우선으로 따른다.
3. 사용자가 `요구하지 않은 변경`은 `하지 않는다`.
4. 요구사항이 `불분명`하면 수정 전에 반드시 `질문`한다.
5. 한글이 깨지지 않도록 주의한다.
6. `테스트 코드 실행` 시에는 `자율`적으로 `실행`한다.
7. `문맥 충돌 시` 줄 단위 `최소 치환`으로 재시도
8. 작성한 사항에 대해 `단계`별로 충분한 `주석`을 작성
9. 단계별로 충분한 `디버깅`용 `로그`를 작성
10. `실행 권한`은 항상 질문하지 말고 `자동`으로 `상승`한다.
11. 모든 `개발`, `수정` 내용과 `변경 내역`은 `Development.md`에 기록한다.
12. 작업 중 발생한 `실수`나 `성과`는 `AGENTS.md`에 기록하여 자체적으로 개선한다.

## 설계 기준
1. 모든 로직 해석의 최우선 기준은 `Entity`이다.
2. 타입, 필드 의미, 값의 형태는 먼저 `Entity`를 기준으로 판단한다.
3. Controller, DTO, Service, Repository는 `Entity` 기준을 따라야 한다.
4. 문서화나 Swagger 수정도 실제 `Entity` 및 `응답 계약`을 바꾸지 않는 선에서 진행한다.

## 수정 원칙
1. `요청받은 범위 안`에서만 `수정`한다.
2. `기존 코드 스타일`, `네이밍`, `계층 구조`를 `유지`한다.
3. 임의의 리팩터링, 추상화, 구조 변경을 하지 않는다.
4. 타입 변경은 실제 도메인 기준이 명확할 때만 진행한다.
5. 값의 형식과 타입을 혼동하지 않는다.
6. `테스트 코드 수정`은 `자율`적으로 한다.
7. 전체 교체는 정말 불가피할 때만, 사전 고지 후 진행
8. 불필요한 수정은 절대 불가. 근거가 명확히 있는 수정만 가능.
9. `부분 패치` 우선

## 응답/문서화 원칙
1. Swagger 문서는 실제 코드와 응답 구조를 기준으로 맞춘다.
2. Swagger를 위해 런타임 계약을 임의로 바꾸지 않는다.
3. 공통 응답 구조를 수정할 때는 전체 사용처 영향을 먼저 확인한다.
4. example, description, schema는 실제 Entity와 DTO 의미에 맞게 작성한다.
5. 도메인별 분류하여 응답한다.

## 검증 원칙
1. 수정 후 가능한 범위에서 컴파일 또는 테스트로 검증한다.
2. 검증하지 못한 경우 그 사실을 명확히 알린다.
3. 실패 시 원인과 영향 범위를 먼저 설명한다.

## 작업 태도
1. 추측보다 코드 확인을 우선한다.
2. 애매하면 일반론보다 현재 프로젝트 구현을 기준으로 답한다.
3. 사용자의 기준이 명확히 제시되면 그 기준을 이후 판단의 우선순위에 반영한다.

## 사후 강평 및 개선 방안 (2026-05-13)

### 1. 주요 실수 사례
*   **컴포넌트 구조 파괴**: `PointRankingDashboard.vue` 수정 시 최상단 `<script setup>` 태그를 누락시켜 빌드 에러를 유발함. 
*   **함수 인터페이스 불일치**: `goToSuccessPage` 호출 시 필수 인자인 `pageHref`를 전달하지 않아 런타임 `TypeError`를 발생시킴.
*   **타입 불일치 간과**: JPA 엔티티의 `OffsetDateTime`과 서비스 로직의 `LocalDateTime` 혼용으로 인한 500 에러 유발.
*   **리팩터링 중 임포트 누락**: Lombok 적용을 위한 `multi_replace_file_content` 사용 시 필수 도메인 클래스(`User`)의 임포트가 누락되어 컴파일 에러 발생.
*   **Lombok 필드 접근 제한 간과**: `@Setter(AccessLevel.NONE)`이 적용된 ID 필드에 대해 테스트 코드에서 `setId()`를 호출하여 컴파일 에러 발생. (해결: `@Builder` 사용)
*   **Mockito 엄격한 스터빙 위반**: 테스트 로직에서 사용되지 않는 메서드를 스터빙하여 `UnnecessaryStubbingException` 발생. (해결: `lenient()` 사용 또는 스터빙 최적화)
*   **통합 테스트 빈 주입 오류**: `@SpringBootTest` 환경에서 `MockMvc` 설정 시 `ObjectMapper` 등 자동 구성된 빈이 `standaloneSetup`에 주입되지 않아 500 에러 또는 `NoSuchBeanDefinitionException`이 발생함. (해결: 서비스 수동 주입 및 MockMvc 설정 보강)

### 2. 성과 및 잘한 점
*   **문서화 자동화**: `Development.md`와 `AGENTS.md`를 실시간으로 업데이트하여 개발 이력의 투명성을 확보함.
*   **Lombok 도입 및 최적화**: 스프링 부트 전 계층에 Lombok을 적용하여 보일러플레이트 코드를 획기적으로 줄이고 유지보수성을 향상함.
*   **프로젝트 전수 검사 및 표준화**: 모든 도메인 엔티티와 주요 서비스/컨트롤러를 전수 조사하여 Lombok 기반 코딩 표준을 적용함.
*   **빌드 및 테스트 자동 검증**: 대규모 리팩터링 후 `./gradlew build` 및 `./gradlew test`를 수행하여 기능의 정합성을 최종 확인함.
*   **Entity 중심 설계**: 모든 계층의 로직을 Entity 기준으로 통일하여 데이터 일관성을 유지함.
*   **정밀한 부분 패치**: 파일 전체 교체 대신 `replace_file_content`를 활용하여 코드 오염을 최소화함.
*   **관리자 전용 데이터 브라우저 구축**: 모든 DB 엔티티를 동적으로 조회할 수 있는 Universal Data Viewer를 구현하여 운영 편의성을 극대화함.

### 3. 개선 및 방지 대책
*   **부분 패치(Partial Patch) 엄수**: 파일 전체 교체보다는 변경이 필요한 코드 블록만 정밀하게 치환하여 구조적 파손을 방지한다.
*   **인터페이스 선제 확인**: 함수나 컴포넌트를 사용하기 전, 해당 정의(Definition)를 반드시 다시 확인하여 인자 누락을 방지한다.
*   **타입 엄격 준수**: 엔티티 클래스에 정의된 데이터 타입을 최우선 기준으로 삼고, 모든 계층에서 이를 통일한다.
*   **임포트 유지 주의**: 대량의 코드 블록을 치환할 때 상단의 임포트 선언부가 의도치 않게 삭제되지 않도록 정밀하게 범위를 지정한다.
*   **표준 SQL/JPA 지향**: 가급적 표준 JPA 메서드를 사용하고, 복잡한 쿼리는 Native Query를 사용하여 호환성 문제를 원천 차단한다.
*   **검증 절차 강화**: 수정 후에는 반드시 해당 기능의 진입점부터 결과 출력까지의 흐름(End-to-End)을 머릿속으로 시뮬레이션하거나 가능한 범위 내에서 테스트를 수행한다.
*   **테스트 지향 설계(Test-Friendly Design)**: 모든 코드는 테스트가 용이하도록 의존성을 분리하고 인터페이스를 명확히 정의하여 작성한다.
*   **문서화 준수**: `Development.md`와 `AGENTS.md`를 개발 프로세스의 필수 단계로 통합하여 기록 누락을 방지한다.
